# design-diff

Локальный инструмент для визуального сравнения референс-дизайна (URL, загруженное изображение
или Figma-фрейм по ссылке) с целевым сайтом по нескольким брейкпоинтам. Находит пиксельные
различия, кластеризует зоны их концентрации (связные области, не сетка), для пары URL↔URL —
даёт постраничный style-diff (палитра/типографика/отступы/layout) и **surgical per-element diff**
(конкретные изменённые/пропавшие/лишние элементы с точными CSS-свойствами и геометрией). По
итогам генерирует готовый промпт для Claude — полностью локально.

Всё выполняется локально. Сетевой доступ: (а) Playwright открывает страницы по URL, которые вы
сами указали, чтобы сделать скриншоты; (б) при выборе источника «Figma» — один запрос к
`api.figma.com` с вашим личным токеном, чтобы скачать рендер нужного фрейма. Никаких других
внешних вызовов, телеметрии и передачи данных куда-либо ещё нет.

## Стек

- Backend: Node 20 + Express + TypeScript (запуск через `tsx`, без шага сборки)
- Скриншоты: Playwright (Chromium)
- Обработка изображений и диффы: `sharp` (все операции с изображениями) + `pixelmatch` (сырой RGBA-дифф)
- Frontend: Vue 3 + Vite + TypeScript + Tailwind CSS 3, тёмная тема
- Валидация: `zod`, загрузка файлов: `multer`, переменные окружения: `dotenv`

```bash
cd "D:\My Projects\design-diff"
npm install
npm run install:browser   # ставит/проверяет Chromium для Playwright (обычно уже закэширован)
npm run dev                # поднимает Vite (5173) и Express (3000) параллельно
```

Откройте http://localhost:5173 — интерфейс проксирует `/api`, `/runs`, `/fixtures` на Express (порт 3000).

### Продакшн-сборка

```bash
npm run build   # собирает web/dist
npm start        # собирает и поднимает Express, который отдаёт web/dist и API на одном порту (3000)
```

## Как пользоваться

1. Выберите источник референса — URL другого сайта, загрузка изображения макета (PNG/JPG) или
   ссылка на Figma-фрейм.
2. Укажите URL целевого сайта, который нужно сверить с референсом.
3. Выберите брейкпоинты (пресеты 1440/1024/768/390 или добавьте свою ширину).
4. При необходимости включите режим "полная страница" (fullPage) и разверните «Расширенные
   настройки» (см. ниже) — для реальных сайтов с cookie-баннерами, анимацией, авторизацией и т.п.
5. Нажмите «Сравнить». Для каждого брейкпоинта появится карточка: оценка совпадения, переключатель
   Side-by-side / Overlay (слайдер до/после) / Diff, список **расхождений по элементам** (что именно
   изменилось — с конкретными CSS-свойствами и сдвигом в px) с подсветкой соответствующих боксов
   поверх скриншота цели, и агрегированная таблица расхождений стилей страницы (только для пары
   URL↔URL).
6. Внизу — готовый промпт для Claude: можно скопировать в буфер или сохранить как `.md`-файл
   и вставить в Claude, чтобы он поправил вёрстку. Промпт построен по элементным расхождениям —
   это самая ценная часть для точечных правок.

## Расширенные настройки (устойчивость к реальным сайтам)

Реальные сайты часто ломают наивный скриншот: cookie-баннеры, ленивая подгрузка, анимации,
страницы за авторизацией. Форма «Расширенные настройки» (все поля опциональны) применяется
одинаково и к референсу-URL, и к целевому сайту:

- **Скрыть элементы перед снимком** (`hideSelectors`) — CSS-селекторы (через запятую или с новой
  строки), которые скрываются (`display:none !important`) перед скриншотом — cookie-баннеры,
  виджеты чата, sticky-полосы.
- **Кликнуть перед снимком** (`dismissSelectors`) — селекторы кнопок вроде «Принять cookies»;
  клик по каждому — best-effort (в try/catch, отсутствие селектора не считается ошибкой).
- **Ожидание загрузки** — `waitUntil` (`load` / `domcontentloaded` / `networkidle`, по умолчанию
  `networkidle`), дополнительная пауза `waitMs` (по умолчанию 500 мс), и опционально —
  `waitForSelector` (ждать появления конкретного элемента, с таймаутом).
- **Замораживать анимации** (`freezeAnimations`, по умолчанию включено) — инъекция CSS,
  отключающего анимации/переходы, плюс эмуляция `prefers-reduced-motion: reduce` — убирает шум
  в диффе от «случайного кадра» CSS-анимации.
- **Авторизация** (`auth`, JSON) — `cookies` (`[{name,value,domain,path}]`), `headers`
  (например `Authorization`) через `context.setExtraHTTPHeaders`, и/или `httpCredentials`
  (`{username,password}`) для HTTP Basic Auth — позволяет снимать страницы за авторизацией.
  Значения остаются локально и никогда не логируются.
- **Снимать только один блок** (`clipSelector`) — вместо полной страницы/вьюпорта снимается
  только bounding box указанного селектора с обеих сторон — удобно, чтобы сравнить один
  компонент/секцию.

## Per-element diff (Feature 2)

Для пары URL↔URL, помимо агрегированного page-level style-diff, инструмент собирает «значимые»
элементы (заголовки h1–h6, `p`, `a`, `button`, `img`, `input`, лендмарки
`header/nav/main/footer/section`, плюс листовые элементы с заметным текстом), сопоставляет
референс↔цель (по точному совпадению текста → похожести текста → структурно по
tag+role+nth-of-type) и считает для каждой пары геометрию (сдвиг/изменение размера в px, порог
значимости — 3px) и посвойственные различия стилей (color, background-color, font-family,
font-size, font-weight, line-height, padding, margin, border-radius, text-align). Элементы, не
найденные в цели — «missing», лишние в цели — «extra». В промпт для Claude попадают только
элементы, которые реально отличаются — это делает чек-лист точечным.

Hot-регионы (зоны концентрации пиксельных различий) считаются как связные компоненты
(8-connectivity, two-pass labeling + union-find) над маской pixelmatch-диффа, а не как сетка —
боксы кластеризуются точно на изменившихся элементах.

## Figma по ссылке (Feature 3)

1. В Figma: `Settings → Security → Personal access tokens → Create new token`, скопируйте токен
   (показывается один раз).
2. Скопируйте `.env.example` в `.env` в корне проекта и впишите `FIGMA_TOKEN=ваш_токен`.
   Файл `.env` уже в `.gitignore` — токен никуда не коммитится и не логируется.
3. В форме выберите источник «Референс — Figma» и вставьте ссылку на фрейм — в идеале с
   `?node-id=...` (в Figma выделите фрейм → «Copy link to selection»). Если `node-id` не указан,
   инструмент попробует взять первый верхнеуровневый фрейм файла, иначе вернёт понятную ошибку.
4. Инструмент запрашивает `GET https://api.figma.com/v1/images/<fileKey>?ids=<nodeId>&format=png&scale=2`
   с заголовком `X-Figma-Token`, скачивает временную PNG-ссылку из ответа и использует её как
   референс — дальше пайплайн идёт как для загруженного изображения (только визуальный дифф, без
   DOM-style-diff — у Figma-рендера нет DOM).

Юнит-тесты парсера ссылок и билдера запроса (без сети, без токена):

```bash
npm run test:figma
```

## Локальные тестовые фикстуры

- `fixtures/a.html` и `fixtures/b.html` — две сознательно различающиеся страницы (цвет, шрифт h1,
  отступы кнопки, ширина контейнера) с одинаковой структурой (header/nav/main/section/footer).
- `fixtures/banner.html` — страница с фиксированным cookie-баннером и CSS-анимацией (пульсирующая
  кнопка) — для проверки `hideSelectors`/`dismissSelectors`/`freezeAnimations`/`clipSelector`.
- `/fixtures/gated` (серверный роут, не статический файл) — отдаёт разный HTML в зависимости от
  заголовка `x-fixture-auth: granted` или cookie `auth=granted` — для проверки, что `auth`
  реально влияет на захваченный контент.

Express отдаёт статические фикстуры по адресам вида `http://localhost:3000/fixtures/a.html` —
удобно для офлайн-проверки пайплайна без выхода в интернет.

## Интеграция с Claude (CLI-гейт + MCP-сервер)

Тот же движок (`server/jobs.ts`) доступен ещё двумя способами, оба переиспользуют пайплайн один в
один и не требуют `ANTHROPIC_API_KEY` — LLM работает только на стороне клиента (Claude
Code/Desktop, ваша подписка), сам инструмент остаётся полностью локальным:

- `npm run verify -- --ref <url> --target <url> [--threshold 90]` — детерминированный gate
  (exit code 0/1/2) для pre-commit хука, CI или скрипта.
- `npx tsx server/mcp.ts` — MCP stdio-сервер с инструментами `compare`/`verify`, которые Claude
  Code/Desktop может вызывать напрямую (`claude mcp add design-diff -- npx tsx server/mcp.ts`).

Подробности, аргументы CLI и пример регистрации `.mcp.json` — в
[`docs/CLAUDE-INTEGRATION.md`](docs/CLAUDE-INTEGRATION.md).

## Известные ограничения

- Матчинг элементов (Feature 2) — жадный O(n·m) по значимым элементам (текст → похожесть текста →
  структурный fallback), с порогом похожести текста 0.5 и минимальной длиной «заметного» текста у
  листовых элементов 8 символов; для очень больших DOM потребовалась бы индексация
  (см. `// shortcut:` в `server/styles.ts`).
- Hot-регионы (Feature 2) — связные компоненты с эвристическими константами (мин. размер блока,
  дистанция слияния соседних боксов) — см. `// shortcut:` в `server/diff.ts`.
- Тальи палитры/типографики/отступов (агрегированный style-diff) — наивный подсчёт по количеству
  элементов (`server/styles.ts`), не по видимой площади на экране.
- Style-diff и per-element diff считаются только для пары URL↔URL — для референса-изображения или
  Figma такого DOM просто не существует, поэтому используются только визуальный дифф и hot-регионы.
- Промпт для Claude собирается по детерминированному шаблону локально — никакого обращения
  к Claude API нет.
- Figma-интеграция (Feature 3) проверена структурно (парсер URL, билдер запроса, ветка
  отсутствующего токена — все через `npm run test:figma` и через реальный `POST /api/compare`).
  Живой рендер реального Figma-файла требует вашего личного `FIGMA_TOKEN` и настоящей ссылки —
  это последний шаг, который может сделать только пользователь.
